## 三套系統各做各的，等於多三套 Excel

很多物流公司在數位轉型時，會分階段導入：先 ERP，再 TMS，再 WMS。每套都單獨上線，每套都跑得起來。

問題是：**單獨能跑，串不起來**。

- ERP 知道客戶下了什麼單，但不知道倉庫實際有多少貨
- WMS 知道倉庫今天出了什麼，但要等夜批才把資料丟給 ERP
- TMS 排好派車，但司機回來後，倉庫的棧板異動沒同步到 WMS

結果就是每天 8 點全公司在做同一件事：**對帳**。倉管打電話給調度，調度問業務，業務問會計，會計問老闆娘。三套系統加起來，比原本的 Excel 還複雜。

這篇文章整理：**為什麼會這樣、4 種常見的整合錯誤、以及一個訂單怎麼正確貫穿三套系統。**

## 三套系統的分工：誰管什麼

先快速釐清角色：

| 系統 | 管什麼 | 單一事實 |
| --- | --- | --- |
| **WMS** | 物理現實：貨在哪、怎麼動 | 倉庫現場狀態 |
| **TMS** | 配送現實：誰開車、走哪、何時到 | 派車與配送狀態 |
| **ERP** | 財務現實：誰欠誰錢、多少成本 | 帳上的事實 |

三套系統如果用對，是**互補**，不是**重疊**。每套系統都有自己的「**唯一事實**」，其他系統不該重複記錄，只能引用。

整合的本質就是：**讓一筆業務交易（一張訂單）能正確地在三套系統間流動，不用人工搬資料**。

## 一張訂單應該怎麼跑

正確的訂單生命週期：

```
1. 客戶下單 → ERP 建立訂單
2. ERP 通知 WMS：請準備出貨
3. WMS 生成揀貨單，按儲位排序給揀貨員
4. 揀貨員手機掃碼揀貨 → WMS 扣帳
5. 出貨完成 → WMS 通知 TMS：請派車
6. TMS 排路線、指派司機 → 司機 LINE App 接單
7. 司機 GPS 即時回報、客戶簽收
8. 配送完成 → TMS 通知 ERP：請開立發票
9. ERP 開發票、應收帳款入帳
10. 客戶付款 → 銀行對帳爬蟲比對 → ERP 沖帳
```

這 10 步全程**零手動 key**。每一步都是「上一步的事件觸發下一步」。

問題不在哪個系統有沒有功能，而在**事件怎麼從 A 系統跑到 B 系統**。

## 常見整合錯誤 1：用人工每天匯出 Excel 對

最常見、最痛、也最沒被當成問題的做法。

每天早上：
- WMS 匯出昨天的出貨明細 Excel
- 老闆娘在 ERP 手動輸入
- 中間一定會有 key 錯、漏 key、key 兩次

這不叫整合，這叫「**用 Excel 當中介**」。三套系統加起來，比原本一套 ERP 還慢。

正確做法：**Webhook 即時推送**。WMS 出貨完成的瞬間，HTTP POST 推給 ERP，ERP 自動開單。

## 常見整合錯誤 2：用排程夜批同步

進階版，但還是不對。

「每晚 12 點 WMS 把當日異動丟給 ERP」這種架構聽起來很合理，但有兩個致命問題：

1. **白天看的數字不準**：業務白天看到的是昨天 12 點的庫存，跟訊號 2 一樣
2. **批次失敗整批爛**：如果某天夜批掛了，沒人知道，第二天才發現整天的資料沒進

正確做法：**事件驅動**。每筆異動發生當下推送，不是累積到夜裡。

## 常見整合錯誤 3：三套系統各自有「客戶主檔」

這是隱形殺手。

ERP 有客戶 1234 號叫「全聯實業」、WMS 有客戶代號 P-001 叫「全聯」、TMS 有客戶 ID #88 叫「全聯股份」。

三邊資料看起來一樣，但**主鍵不一樣**。整合時就要做大量的對應表，每次新增客戶都要在三邊各 key 一次，總有一天會對錯。

正確做法：**單一主檔來源**。客戶主檔放在 ERP（或統一身份系統），WMS 和 TMS 透過 API 即時拉取，永遠不要在子系統重複建客戶。

## 常見整合錯誤 4：沒有「冪等性」

技術詞，但結果很實在。

「冪等性」的意思是：**同一個請求重複送 N 次，結果跟送一次一樣**。

舉例：WMS 出貨完成推 ERP 開發票。如果網路抖一下，WMS 重送了一次，沒有冪等性的 ERP 就會開兩張發票。

物流業每天幾百筆異動，網路、伺服器、重啟，一定會有重送。沒有冪等性，重複付款、重複出貨、重複開票就是時間問題。

正確做法：每筆事件都帶 **唯一事件 ID**，下游系統收到重複 ID 就忽略。Dr.Dow ERP 的「六層付款防呆」中第 5 層就是 API 冪等性，正是為了解決這件事。

## 正確的串接：WMS → TMS → ERP

把三套系統正確串起來後，一個典型的工作日長這樣：

**早上 8:00**
- 業務在 ERP 收到客戶訂單
- ERP 自動建單 → WMS 生成揀貨任務
- WMS 把揀貨任務按儲位排序推到倉管手機

**早上 9:30**
- 倉管掃碼揀貨完成 → WMS 即時扣帳
- WMS 推 TMS：「這批貨準備好了，請排車」
- TMS 自動帶入路線規劃、指派司機

**早上 10:00**
- 司機在 LINE App 看到任務、簽到、出車
- TMS GPS 每 2 分鐘回報位置
- ERP 後台同步看到應收帳款已建立

**下午 3:00**
- 司機回報「客戶簽收」+ 拍照
- TMS 推 ERP：「請開發票」
- ERP 自動產生發票，寄給客戶

**晚上 6:00**
- 銀行收款回來，ERP 銀行爬蟲自動對帳
- 沖銷應收帳款
- 老闆手機收到「今日營收 +XXX 萬」推播

**全天人工 key 的次數：0。**

## 串接不需要砍掉重練

最常見的擔心：「我已經有 ERP 了，要全部換掉嗎？」

**不需要**。Dr.Dow 的做法是：

- **新導入 WMS**：跟既有 ERP 用 API / Webhook 串
- **既有 TMS**：透過事件 bus 跟 WMS / ERP 對話
- **逐個系統替換**：等到舊系統真的卡住再換

不是「全部換 Dr.Dow 才能串」。Dr.Dow WMS / TMS / ERP 設計成可以**單獨上線、漸進整合**：

- 你只要 WMS：先把倉庫掃碼跑起來
- 你想加 TMS：再串派車流程
- 最後再串 ERP：對帳與發票自動化

每加一套，整合成本邊際遞減；每加一套，「人工 key」的工作就少一截。

## 不該追求「百分百自動化」

最後一個重要觀念：**不要追求 100% 自動化**。

整合的目的不是「機器人取代人」，是**讓人不用做機械性的搬資料工作，把心力放在判斷上**。

例如：
- 揀貨複核：機器掃碼防錯出，**人決定**揀錯了怎麼處理
- 銀行對帳：爬蟲抓明細自動比對，**老闆娘決定**異常筆怎麼處理
- 派車路線：AI 規劃路線，**調度員決定**例外狀況怎麼調

整合做到 80~90% 自動，剩下 10~20% 留給人做判斷，就是物流業數位轉型的甜蜜點。

## 下一步

如果你已經有 ERP 想加 WMS、或已經有 TMS 想串 WMS / ERP，建議從這三步開始：

1. **盤點現有系統**：列出 ERP / TMS / 倉庫工具各自記了什麼資料、有沒有重複
2. **找一個最痛的整合斷點**：通常是「白天庫存對不準」或「派車後沒同步財務」
3. **先串一條最痛的線**：不要一次串全部，先把最痛的那條串起來看效果
